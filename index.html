Areas de GIT

Working directory

Es el area de trabajo, donde rara vez Utilizaremos la linea de comandos y donde especialmente estaremos realizando las modificaciones de nuestro proyecto, como la creacion de los archivos, la modidificacion de los mismos e incluso su eliminacion.

  • dir || ls Muestra el contenido de la carperta donde nos encontramos actualmente
  • cd <path Cambia el directorio actual a la ruta especificada
  • mkdir <name crea una carpeta con el nombre indicado
  • git init Inicializa un repositorio de GIT en el directorio actual
  • rm -rf .git este comando elimina el repositorio de git, el parametro -rf indica que debe eliminar la carpeta de manera forzada

Staging Area (INDEX)

Es el Area de control de cambios, donnde se realizan las capturas de pantalla (snapshot) de codigo . Todo contenido que no sea agregado a esta area, sera ignorado por GIT y por tanto no podra recuperarse en caso de perdida. Una vez agregado, GIT esta verificando constantemente si se realizaron modificaciones del mismo.

  • Git status Muestra el estado de los archivos comparados con la ultima confirmacion de cambios, los valores que puede devolver son los siguientes
    • untracked el archivo aun no fue agregado al INDEX y por lo tanto no se le esta esta haciendo seguimiento.
    • Tracked Los cambios fueron agregados al INDEX y estan listos para ser confirmados.
    • modified EL archivo al que se le esta haciendo seguimiento presenta diferencias con respecto a la ultima modificacion
    • commited los cambios realizados fueron registrados por medio de una confirmacion
  • GIT all <file> Agregamos los archivos seleccionados al INDEX para realizarle seguimiento.
  • GIT blame <file> identifica a los autores de las modificaciones de un archivo en particular, siempre y cuando hayan sido confirmadas.
  • git rm --cached <file> Quita del INDEX los cambios almacenados de un archivo en particular, pero mantiene el mismo en el Working directory
  • git restore --staged <file> Restablece los cambios del index de un archivo en particular, respecto de la ultima confirmacion

Repository

Area donde se Almacena la confirmacion de cambios de formato BLOB(Binary Large objetc file) las mismas apuntan a otras confirmaciones atravez de referencias, gracias a estoy es posible navegar dentro de la histora de nuestro proyecto, para ver el estado del mismo durante las diferentes modificaciones.

  • GIT Commit Abre el editor de codigo predeterminado para generar un archivo de confirmacion
  • GIT Checkout <ref> Permite volver a un punto anterior del proyecto segun lo indicado.
  • GIT diff Muestra las diferencia entre el working directory y el ultimo commit
  • GIT remote add <alias> <url> Agrega una dirección como repositorio remoto.
  • GIT Log Muestra el historial de confirmaciones en forma de arbol, con los numeros de hash pertinentes.
  • GIT Push <remote> <url> <branch> Sube los cambios realizados al repositorio remoto.

Apuntadores de GIT

HEAD

Apuntador movil que permite al usuario navegar dentro del historial de cambios mediante referencias, que pueden ser, absolutas como los numeros HASH ej: ABC123F; ; o relativas como HEAD^ o HEAD~1 para desplazarse al commit ancestro o anterior de la posicion actual. Este apuntador nos indica donde nos encontramos dentro del historial de cambios.

BRANCH

Apuntador dinamico que se ubica en el ultimo commit de una historia en particular, podemos crearlas a partir de cualquier confirmacion de cambios, pero todas las ramas creadas comparten algun punto en comun dentro del mismo arbol. La razón por la que existen es, por un lado, facilitar al usuario la navegacion dentro del historial de cambio y, por el otro, para trabajar paralelamente diferentes cambios, experimentales o no, sin afectar la historia principal.
